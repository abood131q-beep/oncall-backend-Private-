# Business Impact Assessment (BIA) — taxi.js: H1 + M1 + H2

**المشروع:** OnCall Backend
**الوحدة المتأثرة:** نظام رحلات التاكسي (`src/routes/taxi.js`)
**التاريخ:** 2026-07-11

---

## 1. الخلاصة التنفيذية

كانت ثغرات H1 و H2 تُمكِّن أي مستخدم مصادَق عليه من انتحال هوية مستخدم آخر في طلبات التاكسي وقبول الرحلات. ثغرة M1 تُتيح إدخال إحداثيات غير صحيحة تُفسد حسابات الأجرة. الإصلاح يُغلق هذه الثغرات الثلاث مع صفر تأثير على وظائف التطبيق.

---

## 2. تحليل المخاطر قبل الإصلاح

### H1: انتحال هوية الراكب

**السيناريو المحتمل:**
مهاجم لديه حساب يُرسل `phone: "رقم_ضحية"` في body طلب التاكسي → الرحلة تُسجَّل على الضحية → رصيد الضحية يُخصم → رسوم التاكسي تُضاف لسجلها.

| البُعد | التأثير |
|-------|---------|
| المالي | خصم رصيد من ضحية دون موافقتها — مسؤولية قانونية |
| الثقة | انكشاف يُدمر ثقة المستخدمين |
| قانوني | انتهاك حماية البيانات الشخصية |
| تشغيلي | سجلات رحلات مُلوَّثة — صعوبة التدقيق |

**الاحتمالية (بدون الإصلاح):** عالية — يكفي أن يعرف المهاجم رقم هاتف ضحية (متاح أحياناً من الـ Socket events).

---

### M1: إحداثيات مزوَّرة

**السيناريو المحتمل:**
مستخدم يُرسل إحداثيات وهمية (مسافة 0 أو مسافة ضخمة) → الأجرة المُقدَّرة خاطئة → غش في الأجرة أو إعطاء خدمة مجانية.

| البُعد | التأثير |
|-------|---------|
| المالي | أجرة 0 على رحلة حقيقية (مسافة صفر) أو رفض سائق لأجرة وهمية كبيرة |
| التشغيلي | بيانات موقع فاسدة في التقارير الإدارية |
| الجودة | تجربة مستخدم سيئة لأجرة مختلفة عن الواقع |

---

### H2: انتحال هوية السائق

**السيناريو المحتمل:**
سائق أ يُرسل `driver_phone: "رقم_سائق_ب"` → الرحلة تُقبَل باسم سائق ب → سائق ب يحصل على الرحلة والعمولة دون علمه → سائق أ لا يُحاسَب.

| البُعد | التأثير |
|-------|---------|
| المالي | سرقة عمولة السائق الشرعي |
| التشغيلي | رحلات تُعيَّن لسائق خاطئ — فوضى لوجستية |
| قانوني | مسؤولية عند حادثة — السجل يُشير لسائق غير صحيح |
| الثقة | سائقون يفقدون الثقة بالمنصة |

## 4. تأثير الإصلاح على الأعمال

### التأثيرات الإيجابية

- **صفر طلبات تاكسي مزوَّرة:** phone مرتبط بـ JWT، لا يمكن تغييره.
- **أجرة دقيقة دائماً:** إحداثيات مُتحقَّق منها قبل الحساب.
- **سجلات موثوقة:** كل رحلة مرتبطة بالسائق الحقيقي المصادَق عليه.
- **حماية Race Condition:** Atomic SQL يمنع قبول رحلة من سائقين في آن واحد.

### التأثيرات على الوظائف الحالية

| الوظيفة | تأثير |
|---------|-------|
| طلب تاكسي (راكب) | ✅ لا تأثير — الـ phone يُستخرج تلقائياً من JWT |
| تقدير الأجرة | ✅ أدق — إحداثيات مضمونة الصحة |
| قبول الرحلة (سائق) | ✅ لا تأثير — phone يأتي من الـ auth token |
| Flutter App | ✅ لا تغيير — الـ app لا يُرسل phone أصلاً في الـ headers (JWT) |
| MCP Tools | ✅ `create_taxi_request` يعمل بشكل طبيعي |

### عدم الانتكاسات

- 54/54 اختبار تمرير (100%) — لا رجعة في الوظائف.
- MCP test: `create_taxi_request` نجح ✅

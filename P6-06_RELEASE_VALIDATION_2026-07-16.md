## 3. اختبار Race Conditions (طلبان متزامنان) — سلامة البيانات ✅ / الاستجابة ⚠️

نُفِّذ فعلياً بـ `Promise.all` مع فحص DB وسجلّ التدقيق بعد كل سيناريو:

| السيناريو | HTTP | حالة DB النهائية | سجلّات التدقيق | سلامة البيانات |
|-----------|:----:|------------------|:--------------:|:--------------:|
| Double Approve | `[200, 500]` | `approved` | **1** APPROVED | ✅ سليمة |
| Double Reject | `[200, 500]` | `rejected` | **1** | ✅ سليمة |
| Approve + Suspend | `[400, 500]` | `approved` | **1** | ✅ سليمة |
| Reactivate + Suspend | `[200, 500]` | `approved` | 3 (شامل الإعداد) | ✅ سليمة |

**نتيجتان مُثبَتتان:**
- ✅ **لا فساد بيانات، لا معالجة مزدوجة، لا سجلّات يتيمة** — كل سيناريو انتهى بحالة صحيحة واحدة مع سجلّ تدقيق واحد للفعل الرابح. `dbTransaction` (BEGIN IMMEDIATE) + فحص التعارض داخل المعاملة يمنعان Double-Approval وLost Update.
- ⚠️ **الطلب الخاسر يُعيد HTTP 500 بدل 409** — بسبب `cannot start a transaction within a transaction` على الاتصال المشترك (سجلّ الخادم يؤكّده). لا يُفسد بيانات، لكنه استجابة خطأ غير لائقة للمشرف الثاني. (نفس الجذر البنيوي لـ C-1، لكن هنا **لا يوجد فساد** لأن كل العمليات داخل المعاملة بلا كتابة سابقة عليها.)

## 5. الأخطاء المكتشفة أثناء التنفيذ

### 🔧 BUG-1 — تظليل مسار `/admin/drivers/pending` — **أُصلح وتُحقِّق منه**
- **الدليل (قبل):** `GET /admin/drivers/pending` → **HTTP 404 «السائق غير موجود»** — التقطه `GET /admin/drivers/:phone` (مُسجَّل أولاً) بـ `:phone="pending"`. قائمة السائقين المعلّقين في لوحة المشرف كانت **غير قابلة للوصول**.
- **الملف:** `src/routes/admin.js` — أُعيد ترتيب المسار الثابت `/pending` قبل المُعامَل `/:phone`.
- **التحقق (بعد):** `GET /admin/drivers/pending` → **HTTP 200، قائمة، count=2** ✓ · بحث السائق المفرد `/:phone` بلا انحدار ✓ · lint 0 · 55/55 وحدة ✓.
- **الخطورة:** متوسطة (كانت تعطّل واجهة اعتماد السائقين).

### ⚠️ BUG-2 — `toggle` القديم لا يُفعّل دخول السائق — **مُبلَّغ (قرار تصميم)**
- **الدليل:** toggle-then-login فشل **3/3** (`status=pending`). `PUT /admin/drivers/:phone/toggle` يضبط `is_active=1` فقط، بينما دخول السائق يعتمد `approval_status` (P6-06). فتفعيل سائق pending عبر toggle لا يُمكّنه من الدخول.
- **الملف:** `src/routes/admin.js` (toggle) مقابل `src/routes/auth.js:132` (بوابة approval_status).
- **لماذا لم يُصلَح تلقائياً:** «الإصلاح» قرار تصميم (حذف toggle للسائقين أم مزامنته مع approval_status). المسار الصحيح `/approve` يعمل تماماً. يحتاج قراراً منك.
- **الخطورة:** منخفضة–متوسطة (endpoint قديم مُضلِّل؛ سبَّب فشل اختبار التكامل رقم 4).

### 🔴 FINDING — تعارض متزامن يُعيد 500 بدل 409 — **مُبلَّغ (جذر بنيوي مشترك)**
- الطلب الخاسر في سيناريوهات Race يُعيد 500 (`cannot start a transaction within a transaction`). لا فساد بيانات، لكن استجابة غير لائقة. نفس جذر C-1 (اتصال SQLite واحد). الإصلاح بنيوي (اتصال-لكل-معاملة / تسلسل / Postgres) لا ترقيع نقطة بنقطة.

### 🔴 C-1 (من مراحل سابقة) — لا يزال قائماً — **غير مُصلَح في هذه الجولة**
- إكمال رحلتين متزامناً → 500 + حالة دفع غير متسقة (`completeTrip` يُلتزَم قبل معاملة الدفع). مُثبَت حياً سابقاً. **حاجب إطلاق.**

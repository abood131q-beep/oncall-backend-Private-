## 0. Purpose & Authority

This document is the **single Source of Truth** for how any Enterprise Kernel (ADR-016 …
ADR-041) is integrated into the OnCall backend. From ratification onward, **every** kernel
integration MUST comply with this standard. It exists to eliminate duplication across future
phases: a phase document should describe only what is *kernel-specific* and reference this
standard for everything common.

**Reference implementation.** The rules below are not aspirational — they are the distilled,
proven pattern already shipped and verified in:

- **Configuration shadow** (Phase 17.3): `src/platform-adapters/configuration/*`,
  `src/enterprise/configShadow.js` — parity 100%.
- **Observability shadow** (Phase 17.4): `src/platform-adapters/observability/*`,
  `src/enterprise/observabilityShadow.js` — parity 100%.

Both sit on the Hosted Service + Adapter Layer from Phase 17.2 (`src/hosted-service/`,
`src/platform-adapters/`, `src/enterprise/`).

**Normative language.** MUST / MUST NOT / SHOULD / MAY follow RFC-2119. A violation of any
MUST requires an **approved ADR** (see §11).

## 1. Shadow Principles (MUST)

A shadow integration runs an Enterprise Kernel *beside* the legacy system for comparison only.
The shadow:

1. MUST NOT change runtime behavior.
2. MUST NOT become authoritative — the legacy system always produces the served result.
3. MUST NOT block requests (comparison runs out-of-band / fire-and-forget).
4. MUST NOT throw to callers — every kernel/adapter error is caught and recorded.
5. MUST NOT modify application state.
6. MUST NOT modify persistent state (the kernel owns no app data in shadow mode).
7. MUST always return the legacy result.
8. SHOULD execute asynchronously wherever possible; where a synchronous surface is required
   (e.g. `shadowGet`), the comparison MUST still be non-blocking or fire-and-forget.
9. MUST remain fully replaceable / removable via feature flags with zero residue.

**Canonical shape** (both reference shadows implement this):

```
legacyResult = legacy.read()                       # authoritative
if (!enabled() || !adapter.consumed()) return legacyResult   # short-circuit, no kernel touch
try   kernelResult = adapter.read()                # ONLY kernel touch, via the adapter
catch recordVerificationFailure(); return legacyResult       # never throw
compare(legacyResult, kernelResult); recordMetrics()
return legacyResult                                # ← unconditional, authoritative
```

## 2. Adapter Principles (MUST)

The Adapter Layer (`src/platform-adapters/<kernel>/`) is the ONLY boundary permitted to touch a
kernel. Adapters:

1. MUST be **stateless** (no mutable instance state beyond a monotonic id counter used for
   fresh-component isolation, which is not observable state).
2. MUST be **deterministic** — same input ⇒ same translation.
3. MUST be **reversible** whenever a decode is defined (`encode ∘ decode = identity`).
4. MUST be **lossless** where parity requires it (use codecs for categorical/boolean values;
   see the observability `check.<n>`/`readiness`/`liveness` gauge codec).
5. MUST own **translation only** — shape/name mapping between legacy and kernel.
6. MUST NOT own business logic.
7. MUST NOT own persistence.
8. MUST NOT call another adapter directly (adapters are siblings; composition happens in
   `src/enterprise/`).
9. MUST NOT cache mutable state.
10. MUST be **inert without a port**: constructed with no injected kernel port, `consumed()`
    returns `false` and every active method rejects with `AdapterNotWiredError`
    (`src/platform-adapters/_base.js`).
11. MUST NOT expose any repository/DB/persistence surface. (Enforced by the adapter-layer test
    that forbids member names matching `/repo|repository|db|sqlite|database/i` — name adapter
    methods accordingly, e.g. `toKernelSpec`, `readComponent`, not `toReport`/`readback`.)

## 3. Feature Flag Standard (MUST)

Every kernel integration MUST expose exactly two flags:

| Flag | Meaning |
|---|---|
| `PLATFORM_<KERNEL>` | Compose/seed the kernel and inject its port into the adapter. |
| `SHADOW_<KERNEL>` | Additionally run parity comparisons. |

Rules:

1. `SHADOW_<KERNEL>` **requires** `PLATFORM_<KERNEL>`; a flag selector MUST downgrade
   `shadow=true, platform=false` → `shadow=false` (reference: `selectConfigFlags`,
   `selectObservabilityFlags`).
2. Flags MUST default to `0` (OFF).
3. With **both OFF**, boot MUST be **byte-identical to the previous phase** — proven by test.
4. Flags MUST be strict string `'1'` to enable (anything else = OFF).
5. Flags MUST be documented in `.env.example` with defaults + rollback meaning.
6. Flags MUST be independent across kernels (enabling one MUST NOT affect another; reference:
   config and observability shadows run together, both 100%).

`<KERNEL>` is the uppercased kernel name, e.g. `CONFIG`, `OBSERVABILITY`, `IDENTITY`.

## 4. Parity Verification Standard (MUST)

Every kernel integration MUST define, in its `03 Parity Report`:

1. **Verification Categories** — the exhaustive list of value families compared (e.g. config:
   env vars, defaults, missing, booleans, numerics, arrays, objects; observability: health,
   readiness, liveness, counters, gauges, timers, tags, event/log metadata).
2. **Comparison Strategy** — deep structural equality (primitives, arrays, plain objects,
   `NaN`), with presence parity where "missing" is meaningful. Reuse the standard `deepEqual`.
3. **Parity Formula** — `parityPct = matches / comparisons × 100` (100 when `comparisons == 0`).
4. **Mismatch Rules** — every mismatch is recorded with a descriptor; **sensitive keys MUST be
   redacted** (values omitted; only types recorded). Reference regex:
   `/secret|token|api[_-]?key|apikey|password|account|firebase|credential/i`.
5. **Failure Rules** — any kernel/adapter error increments `verificationFailures`, is recorded
   (redacted), and the legacy result is returned. A failure MUST NOT throw or block.
6. **Confidence Calculation** — see §5.
7. **Coverage Measurement** — `coveragePct` = fraction of the kernel's declared verification
   surface actually exercised by the shadow (see §5).

**Target:** parity MUST reach **100%** before promotion past *Verified* (§10). Where the kernel
cannot natively represent a legacy value, the adapter MUST provide a lossless codec so parity
is exact — it MUST NOT lower the bar by ignoring fields.

## 5. Shadow Metrics Standard (MUST)

Every integration MUST maintain **isolated, in-memory** shadow metrics, separate from the
application's own metrics/`/metrics` endpoint. Minimum fields:

| Metric | Definition |
|---|---|
| `requests` | shadow reads/passes initiated |
| `comparisons` | per-field comparisons performed |
| `matches` | comparisons that matched |
| `mismatches` | comparisons that differed |
| `verificationFailures` | kernel/adapter errors during a pass |
| `latency` | `{ samples, avgMs, maxMs }` of comparison latency |
| `parityPct` | `matches / comparisons × 100` |
| `confidenceLevel` | `matchRatio × min(1, comparisons / N)` — low sample counts cap confidence (reference `N = 20`); `0` when no comparisons |
| `coveragePct` | declared verification surface actually exercised (`checked / declared × 100`) |
| `mismatches_log` | bounded ring buffer of (redacted) mismatch/failure descriptors |

Rules:

1. Recording a metric MUST have **no** runtime effect.
2. Metrics MUST NOT be wired into the application's `/metrics` output.
3. `mismatches_log` MUST be bounded (reference limit 100) and MUST redact sensitive values.

> **Note (backfill):** `coveragePct` is newly mandated by G1.0. The 17.3/17.4 shadows already
> track every other field; they satisfy this standard except for the explicit `coveragePct`
> counter, which future integrations MUST include and existing ones MAY backfill when next
> touched (this standard does not itself modify shipped integrations — see Deliverables).

## 6. Rollback Standard (MUST)

Every integration MUST provide, in its `04 Rollback Guide`:

1. **Rollback Levels** — most-surgical → broadest (e.g. `SHADOW_<KERNEL>=0` →
   `PLATFORM_<KERNEL>=0` → `PLATFORM_HOST=0` → `PLATFORM_ENABLED=0`).
2. **Rollback Safety Matrix** — a table with columns **Rollback Level · Service Impact · User
   Impact · Data Risk · Downtime**. In shadow mode every row MUST read `None / None / None /
   Restart only` (reference: `architecture/phase-17.4/04_..._ROLLBACK_GUIDE.md §3`).
3. **Rollback Verification** — commands/assertions proving the rollback took effect (adapter
   inert, parity report `null`, endpoints unchanged).
4. **Incident Runbook** — numbered steps for on-call.

Rollback MUST require **feature flags only** — no code change, no redeploy of different code,
no migration.

## 7. Testing Standard (MUST)

Every integration MUST include:

| Test type | Proves |
|---|---|
| **Unit Tests** | adapter translation/codec, legacy source, metrics |
| **Integration Tests** | full enterprise boot wiring (host + adapter + shadow) |
| **Parity Tests** | 100% parity across all declared categories |
| **Shadow Tests** | disabled ⇒ legacy returned + no kernel interaction; enabled ⇒ compares |
| **Failure Tests** | kernel throws ⇒ recorded, legacy returned, never throws |
| **Flag Tests** | `SHADOW` requires `PLATFORM`; strict `'1'` semantics |
| **Regression Tests** | prior phases stay green; adapter-surface guard holds |
| **A/B Tests** | legacy vs shadow-enabled HTTP responses byte-identical (`tests/integration/<kernel>-shadow-ab.mjs`, auto-discovered by `scripts/run-ab.mjs`, prints `Result: IDENTICAL`) |

Constraints:

1. Adapters and shadows MUST be testable **without** the DB-backed application (inject fakes),
   so unit/parity tests run anywhere. A/B tests MAY require the real runtime (sqlite) and run
   on the app's OS / CI.
2. The full unit suite MUST pass with `--test-force-exit` and the CI lint gate
   (`eslint 'src/**/*.js' server.js database.js --max-warnings 0`).

## 9. Operational Readiness (MUST)

Before an integration is declared complete, it MUST prove:

1. **100% parity** (0 mismatches, 0 verification failures) across all declared categories.
2. **Rollback tested** (both-off ≡ previous phase; flag-only reversal verified).
3. **No runtime behavior change.**
4. **No API change.**
5. **No startup change.**
6. **No shutdown change.**
7. **No persistent data ownership** by the kernel.
8. **No user-visible behavior change** (Flutter unaffected).

Evidence MUST be captured in `03 Parity Report` (in-environment results + the A/B gate to run
on the app OS / CI).

## 10. Promotion Rules (Kernel Lifecycle)

A kernel advances through these states; each transition is explicit and gated:

```
Disconnected → Connected → Shadow → Verified → Production Shadow → Candidate Ownership → Authoritative
```

| State | Meaning |
|---|---|
| Disconnected | adapter inert; kernel not composed for this concern |
| Connected | `PLATFORM_<KERNEL>=1`; port injected; no comparisons |
| Shadow | `SHADOW_<KERNEL>=1`; comparisons run out-of-band |
| Verified | 100% parity in-environment + unit/parity tests green |
| Production Shadow | soak in production with acceptable confidence + coverage, zero drift |
| Candidate Ownership | A/B identical + rollback verified; ADR proposed to let kernel own the concern |
| Authoritative | **requires a separate approved ADR**; outside the scope of shadow standard |

Promotion **past Verified** requires ALL of:

- parity = 100%;
- zero verification failures;
- A/B identical (`Result: IDENTICAL`);
- rollback verified;
- `confidenceLevel` ≥ the kernel's declared acceptance threshold;
- `coveragePct` ≥ the kernel's declared acceptance threshold.

Reaching **Authoritative** (kernel becomes the source of truth) is explicitly **out of scope**
for shadow integrations and MUST be authorized by its own ADR — it typically also depends on a
byte-compatible DB-backed provider (see the Phase 17.1 readiness blockers B1/B2).

## 11. Architectural Invariants (Permanent)

These are permanent and MUST NEVER be violated without an **approved ADR**:

1. The legacy system is authoritative until an ADR promotes a kernel to *Authoritative*.
2. Only the Adapter Layer communicates with kernels; no application module imports a kernel.
3. Shadows never change runtime, never throw to callers, never block, never own persistent
   state.
4. Every kernel coupling is flag-gated and reversible with flags alone.
5. Both flags OFF ≡ the previous phase, byte-for-byte.
6. Adapters are translation-only, stateless, deterministic, and expose no persistence surface.
7. Shadow metrics are isolated and never affect runtime.
8. Each integration ships the exact six documents in §8 plus the tests in §7.

## 13. Governance

This chapter turns G1.0 from an engineering guide into an institutional standard by defining
who may change it and through which process.

### 13.1 Ownership

- **Standard Owner:** Principal Engineering. The Owner is accountable for the integrity of the
  standard and is the approver of record for every change.
- **Custodian(s):** delegated maintainers (senior platform engineers) who may triage and
  prepare changes but not ratify MAJOR changes alone.
- **Contributors:** any engineer may propose a change via Pull Request.

### 13.2 Change classes → required process → version bump

| Change class | Examples | Required process | Version |
|---|---|---|---|
| **Editorial** | typos, wording, new examples, clarifying a rule without changing its meaning | PR + 1 reviewer (Custodian) | PATCH (`1.0.x`) |
| **Additive normative** | add a new MUST/SHOULD that does not conflict with existing rules; tighten an existing rule; add a metric/checklist item | PR + **Owner approval** | MINOR (`1.0 → 1.1`) |
| **Breaking normative** | relax/remove a MUST; change a Feature-Flag or Promotion gate; alter an **Architectural Invariant (§11)** | **Approved ADR** + Owner sign-off | MAJOR (`1.x → 2.0`) |

### 13.3 When an ADR is required (not just a PR)

An ADR is **mandatory** — a Pull Request alone is insufficient — for any of:

1. Violating, relaxing, or removing an **Architectural Invariant (§11)**.
2. Changing the **Promotion Rules (§10)** or their gates, or promoting any kernel to
   **Authoritative** (kernel becomes source of truth).
3. Changing the **Feature-Flag contract (§3)** (naming, defaults, or the SHADOW⊂PLATFORM rule).
4. Allowing an adapter to hold business logic, persistence, or authoritative behavior.
5. Any MAJOR change in §13.2.

The ADR MUST reference this standard and, on approval, the standard is re-issued at the next
MAJOR version with the ADR recorded in the version header (`Ratifying ADR`) and history.

### 13.4 When a Pull Request suffices

A PR (no ADR) is sufficient for **Editorial** and **Additive normative** changes (§13.2) that
do not touch §13.3. The PR MUST: update the **Standard Version** header, add a one-line entry
to the version history, and — for additive rules — extend the §0 per-integration checklist if
relevant.

### 13.5 When a new version must be issued

- **PATCH** on every merged Editorial PR (optional to publish; batch allowed).
- **MINOR** whenever an Additive normative change merges.
- **MAJOR** whenever a Breaking normative change is ratified by ADR.
Superseded versions are retained in version control for history; the header always names the
current active version and its ratifying ADR.

### 13.6 Compliance & enforcement

- Every new integration's §8 documents MUST include the §0 compliance checklist, filled.
- CI enforces the mechanical invariants (lint gate; adapter-surface guard; both-flags-off ≡
  previous phase; A/B `Result: IDENTICAL`). A failing gate blocks promotion past *Verified*
  (§10) regardless of documentation status.
- Exceptions require a time-boxed waiver approved by the Owner **and** a tracking ADR.
